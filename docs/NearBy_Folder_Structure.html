Naming Convention Rules

Backend Files (Node.js)

  • routes/orders.jsSingular noun, lowercase, .js — one file per domain
  • services/cashfree.jsNamed after the service/tool it wraps, lowercase
  • jobs/autoCancel.jscamelCase verb phrase — describes what the job does
  • jobs/notifyShop.jscamelCase verb+noun — action + target
  • middleware/roleGuard.jscamelCase — describes what it guards/does
  • socket/gpsTracker.jscamelCase — socket event handler group
  • utils/idempotency.jscamelCase — utility concept name
  • src/index.jsAlways index.js for entry points

React Native Files (TypeScript)

  • HomeScreen.tsxPascalCase + Screen suffix for all screens
  • ShopProfileScreen.tsxDescriptive PascalCase + Screen
  • OrderTrackingScreen.tsxNever "Page", always "Screen" in RN
  • ShopCard.tsxPascalCase for reusable components, no suffix
  • TrustBadge.tsxPascalCase, describes what it renders
  • useOrder.tscamelCase with "use" prefix for hooks
  • useSocket.tsuse + noun for all custom hooks
  • cartStore.tscamelCase + Store for Zustand stores
  • api.tsSingle api.ts service in each app

Documentation Files

  • CLAUDE.mdSCREAMING_SNAKE → read by AI. Always in root.
  • README.mdSCREAMING_SNAKE → standard convention
  • SPRINT_PLAN.mdSCREAMING_SNAKE → important human docs
  • CODING_CONVENTIONS.mdSCREAMING_SNAKE → team standards docs
  • API_CHANGELOG.mdSCREAMING_SNAKE → changelog docs
  • ADR-001-cashfree.mdADR-NNN-kebab-case — sequential, topic in name
  • ADR-010-monorepo.mdPad to 3 digits for sort order
  • PRD.htmlUPPER for major deliverables (PRD, FLOWS, API)

Database Migration Files

  • 001_profiles.sqlNNN_snake_case — 3-digit prefix for order, table name
  • 004_orders.sqlOrder matters — FK deps must run in correct sequence
  • 009_rls_policies.sqlRLS always last — tables must exist first

Git Branch Names

  • feature/BE-order-auto-cancelfeature/SCOPE-kebab-description
  • feature/CA-07-cart-managementCA=Customer App, SA=Shop, DA=Delivery, BE=Backend
  • fix/otp-lockout-redisfix/kebab-description of the bug
  • hotfix/cashfree-webhookhotfix/ prefix for production emergencies only
  • chore/update-dependencieschore/ for non-feature non-fix work
  • docs/add-adr-011docs/ for documentation-only PRs

Environment Variables

  • SUPABASE_URLSCREAMING_SNAKE — all env vars always
  • CASHFREE_APP_IDPREFIX_VARIABLE — service name prefix
  • JWT_SECRETCATEGORY_NAME pattern
  • EXPO_PUBLIC_API_URLEXPO_PUBLIC_ prefix for RN client-safe vars
  • ORDER_ACCEPT_WINDOW_SECONDSDomain + setting + unit

How to Use This Repo with Claude / Cursor / Copilot

Follow this guide every time you start a new AI coding session to prevent context loss.

Step 1 — Always paste CLAUDE.md first

# At the start of every Claude conversation, say:

"Here is my project context file. Read it before helping me."

# Then paste the entire contents of CLAUDE.md
# Claude will now know: full stack, all domain rules,
# what never to do, current build status.

Step 2 — Then paste the relevant file

# If working on order flow:
"I'm working on backend/src/routes/orders.js"
# Paste the current file contents

# If working on a new file:
"I need to create backend/src/jobs/autoCancel.js"
# Claude knows the pattern from CLAUDE.md

Step 3 — Reference specific docs by name

# Claude knows the doc structure. You can say:
"Check EDGE_CASES.md — how should I handle the
 'shop doesn't respond in 3 minutes' case?"

"What does ADR-001 say about our payment gateway choice?"

"I'm on Sprint 3, Task 3.5 — implement the auto-cancel job."

# Claude can reference these because CLAUDE.md
# explains what each document contains.

Cursor / VS Code setup

# In Cursor, add CLAUDE.md as a "rules" file:
# Settings → Rules for AI → paste CLAUDE.md content

# Or create .cursorrules in repo root:
cp CLAUDE.md .cursorrules

# Cursor will read this automatically for every chat
# in this workspace — no manual pasting needed.

What CLAUDE.md gives Claude instantly

Full tech stack Why each tool chosen All domain rules API conventions Socket.IO rooms BullMQ queue names DB table names Error code list 12 "never do" rules Current build status Folder structure How to ask for help

Update CLAUDE.md after every sprint

# Every Friday, update this table in CLAUDE.md:

| Module         | Status | Notes              |
|----------------|--------|--------------------|
| Auth (OTP+JWT) | ✅ Done | Tested, deployed   |
| Shop CRUD      | ✅ Done | KYC upload working |
| Order flow     | 🔵 WIP  | Sprint 3, 60% done |
| Payment        | ⬜ Next | Sprint 4           |

# This means every new AI session knows exactly
# what's built and what's next — zero context rebuild.

First Commands — Run In This Exact Order

Day 1 setup. Run these once to create the entire project skeleton.

1. Create GitHub repo & scaffold folders

# Create org + repo on GitHub first, then:
git clone https://github.com/nearby-app/nearby.git
cd nearby

# Run the scaffold script (from GITHUB_STRUCTURE.md)
mkdir -p apps/{customer,shop,delivery,admin}/src
mkdir -p apps/_shared/{components,hooks,constants}
mkdir -p backend/src/{routes,middleware,services,jobs,socket,utils}
mkdir -p supabase/migrations
mkdir -p docs/ADR
mkdir -p .github/workflows

# Create all placeholder files
touch CLAUDE.md README.md docker-compose.yml .env.example .gitignore
touch docs/{PRD.html,ARCHITECTURE.md,API.md,FLOWS.md,EDGE_CASES.md}
touch docs/{SPRINT_PLAN.md,SETUP.md,CODING_CONVENTIONS.md,API_CHANGELOG.md,SUPPORT_RUNBOOK.md}
touch .github/PULL_REQUEST_TEMPLATE.md
touch .github/workflows/ci.yml

2. Populate the critical files first

# Order matters. Do these before any code.

1. Copy CLAUDE.md content → nearby/CLAUDE.md
2. Copy README.md content → nearby/README.md
3. Copy .env.example content → nearby/.env.example
4. Copy docker-compose.yml → nearby/docker-compose.yml
5. Copy all docs/*.md files → nearby/docs/
6. Copy ADR.md content → nearby/docs/ADR/ (split per ADR)
7. Copy PRD.html → nearby/docs/PRD.html

# Only THEN start writing code

3. Git init + first commit

git add .
git commit -m "chore(init): scaffold monorepo structure and all docs"
git push origin main

# Create branches immediately
git checkout -b develop
git push origin develop

git checkout -b staging
git push origin staging

# All future work starts from develop
git checkout develop
git checkout -b feature/BE-sprint1-infrastructure

4. Set up Cursor rules (one time)

# Makes Claude/Cursor read CLAUDE.md automatically
cp CLAUDE.md .cursorrules

# Add to .gitignore if you don't want it committed:
echo ".cursorrules" >> .gitignore

# OR commit it (recommended) so all team members benefit
git add .cursorrules
git commit -m "chore(ai): add cursorrules for AI context"

5. Start Sprint 1 (backend)

# Initialize backend Node.js project
cd backend
npm init -y
npm install express socket.io @supabase/supabase-js ioredis bullmq \
  typesense @aws-sdk/client-s3 jsonwebtoken bcrypt \
  joi winston dotenv sharp multer uuid express-rate-limit \
  helmet cors

npm install -D nodemon jest supertest eslint

# Create package.json scripts
# "dev": "nodemon src/index.js"
# "test": "jest --coverage"
# "lint": "eslint src/"
# "seed": "node scripts/seed.js"

# Start dev server
cd .. && docker-compose up -d redis typesense
cd backend && npm run dev

6. Initialize React Native apps

# Customer app
cd apps/customer
npx create-expo-app . --template expo-template-blank-typescript
npm install @tanstack/react-query zustand axios socket.io-client \
  expo-location expo-camera expo-image-picker expo-secure-store \
  expo-notifications @react-navigation/native @react-navigation/stack

# Repeat for shop and delivery apps
# Admin dashboard
cd apps/admin
npm create vite@latest . -- --template react-ts
npm install @tanstack/react-query axios tailwindcss

7. Daily workflow from here

# Start of every day
git checkout develop && git pull origin develop
git checkout -b feature/BE-[task-from-sprint-plan]

# Write code → test → commit often
git commit -m "feat(orders): implement auto-cancel job"

# End of day / feature complete
git push origin feature/BE-[task]
# Open PR on GitHub → assign reviewer

# Friday ritual
# 1. Update SPRINT_PLAN.md status column
# 2. Update CLAUDE.md build status table
git add CLAUDE.md docs/SPRINT_PLAN.md
git commit -m "chore(sprint): week N progress update"
