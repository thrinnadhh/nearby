Step 1 — Install Node.js 20+
Required for the backend API and tooling. Use nvm to avoid permission errors.
→
Mac / Linux — install via nvm (recommended)
macOS · Linux
nvm lets you switch Node versions and avoids all permission issues. Never use sudo with npm.
Terminal
# Install nvm (Node Version Manager)
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.1/install.sh | bash

# Reload your shell config
source ~/.zshrc    # Mac with zsh (default)
# OR
source ~/.bashrc   # Linux / older Mac

# Install Node.js 20 LTS
nvm install 20
nvm use 20

# Verify — should show v20.x.x
node --version
npm --version
→
Windows — install via installer
Windows
PowerShell (run as Administrator)
# Option A: Use winget
winget install OpenJS.NodeJS.LTS

# Option B: Download from nodejs.org → LTS version
# Run the .msi installer, check "Add to PATH"

# Open NEW PowerShell window after install

Windows users: also install Git for Windows from git-scm.com — Claude Code requires it internally.

Step 7 — Create NearBy Custom Slash Commands
These turn your docs into one-tap Claude Code instructions. Run this in terminal (not inside claude).
→
Create all 5 commands at once
# Run from the nearby/ root directory (normal terminal, not inside claude)

cat > .claude/commands/start-task.md << 'EOF'
Read SPRINT_PLAN.md. Find the first task marked ⬜ in the current sprint.
Tell me: task number, description, which architecture section covers it,
which edge cases apply from EDGE_CASES.md, and the exact file to create/edit.
Change its status to 🔵 in SPRINT_PLAN.md.
EOF

cat > .claude/commands/review-code.md << 'EOF'
Review the current file against:
1. CODING_CONVENTIONS.md (naming, patterns, error codes)
2. Security checklist: no secrets, server-side pricing, HMAC on webhooks,
   JWT+roleGuard on all routes, Joi validation, rate limiting applied
3. EDGE_CASES.md — are all failure modes handled?
4. API.md — does response format match the contract?
List every issue found. Then fix them all.
EOF

cat > .claude/commands/finish-task.md << 'EOF'
1. Mark the task I just completed as ✅ in SPRINT_PLAN.md
2. Update CLAUDE.md build status table for the completed module
3. Log any new/changed endpoints in API_CHANGELOG.md
4. If new edge cases were discovered, add them to EDGE_CASES.md
5. Show a summary of what was completed this session
EOF

cat > .claude/commands/check-edges.md << 'EOF'
Read EDGE_CASES.md. Find every edge case for the feature I am working on.
For each: show the scenario, how current code handles it, whether it matches
the spec, and the fix if it doesn't match.
EOF

cat > .claude/commands/sprint-status.md << 'EOF'
Read SPRINT_PLAN.md and show:
- Current sprint number and its goal
- Tasks: ✅ done / 🔵 in-progress / ⬜ not started / 🔴 blocked
- Percentage complete
- What is blocking progress
- Recommended next task
Also check CLAUDE.md build status and flag any inconsistencies.
EOF

git add .claude/
git commit -m "chore(claude): add NearBy custom slash commands"

Sprint 1 — Auth Prompt
After the kickoff scaffold is complete, use this to build the OTP authentication system. This is Sprint 1, Tasks 1.13–1.17.
🔵 Sprint 1 · Auth · Tasks 1.13–1.17
Build the complete authentication system for NearBy. Sprint 1, Tasks 1.13–1.17. Before writing any code, read: - CLAUDE.md → Auth & Authorization section - EDGE_CASES.md → OTP brute force and rate limiting scenarios - API.md → /auth/send-otp, /auth/verify-otp, /auth/refresh, /auth/me endpoints Build these files in order: 1. backend/src/services/msg91.js - sendOtp(phone, otp) function using MSG91 API - In NODE_ENV=development: log OTP to console, don't send real SMS - Use MSG91_AUTH_KEY and MSG91_TEMPLATE_OTP from .env 2. backend/src/middleware/rateLimit.js - Redis-backed rate limiting factory - OTP: max 5 requests per phone per hour - Orders: max 10 per customer per hour - Global: 100 requests per IP per minute 3. backend/src/middleware/roleGuard.js - Factory function: roleGuard(['customer', 'admin']) - Reads role from req.user (set by auth middleware) - Returns 403 FORBIDDEN if role not in allowed list 4. backend/src/middleware/auth.js - Verify JWT from Authorization: Bearer header - Attach req.user = { userId, phone, role, shopId } - Return 401 UNAUTHORIZED if no token or invalid 5. backend/src/routes/auth.js - POST /auth/send-otp: validate phone, generate 6-digit OTP, store in Redis with 5-min TTL, call msg91.sendOtp() - POST /auth/verify-otp: check OTP in Redis, lockout after 3 wrong attempts (10-min lockout), return JWT on success - POST /auth/refresh: validate refresh token, issue new access token - GET /auth/me: return req.user profile from Supabase 6. backend/src/utils/validators.js - sendOtpSchema: phone required, Indian mobile (10 digits) - verifyOtpSchema: phone + otp (6 digits) required - Use Joi for all validation Critical rules: - OTP: exactly 6 digits, 5-minute TTL in Redis - Wrong OTP → increment attempt counter in Redis - 3rd wrong attempt → lock phone for 10 minutes, return OTP_LOCKED - SMS rate limit → 5 OTP requests per phone per hour → OTP_RATE_LIMITED - Dev mode (NODE_ENV=development): OTP is always 123456 After building, write tests in backend/src/routes/auth.test.js: - Happy path: send OTP → verify → get JWT with correct role - 3 wrong attempts → account locked for 10 minutes - 4th attempt during lockout → OTP_LOCKED error - OTP expiry after 5 minutes → OTP_EXPIRED error - Rate limit after 5 requests → OTP_RATE_LIMITED error Mark tasks 1.13, 1.14, 1.15, 1.16, 1.17 as ✅ in SPRINT_PLAN.md when done.

GET /search/shops: Typesense geo search, params: lat, lng, radius_km, category, open_only - GET /search/products: Typesense text search, params: q, category, shop_id Critical rules: - PATCH /shops/:id/toggle: Typesense update must happen within 15 seconds - Sprint 3 — Order Engine Prompt
The most critical module. Read all edge cases before starting. Sprint 3, Tasks 3.1–3.14.
🟢 Sprint 3 · Orders · Tasks 3.1–3.14 (CRITICAL MODULE)
Build the Order flow. Sprint 3, Tasks 3.1–3.14. This is the most critical module — read everything before writing code. Read ALL of these before starting: - CLAUDE.md → Key Domain Rules → Orders section (every single rule) - ARCHITECTURE.md → Order lifecycle flow diagram - EDGE_CASES.md → ALL 5 order-related edge cases — every single one - API.md → all /orders endpoints - supabase/migrations/004_orders.sql → exact column names - supabase/migrations/005_order_items.sql → order_items structure Build in this exact order: 1. backend/src/utils/idempotency.js - checkIdempotencyKey(key): returns existing result if key seen before - setIdempotencyKey(key, result, ttlSeconds): stores key + result in Redis - TTL: 600 seconds (10 minutes) 2. backend/src/services/fcm.js - sendPush(token, title, body, data): send single FCM notification - sendBatchPush(tokens, title, body, data): send to multiple devices - Use Firebase Admin SDK with FIREBASE_PROJECT_ID, FIREBASE_PRIVATE_KEY 3. backend/src/jobs/notifyShop.js - BullMQ queue: 'notify-shop' - Worker: send FCM high-priority push to shop owner + MSG91 SMS as fallback - Payload: { orderId, shopId, customerName, itemCount, total } 4. backend/src/jobs/autoCancel.js - BullMQ queue: 'auto-cancel' - Delayed job: fires ORDER_ACCEPT_WINDOW_SECONDS after order creation - Worker MUST check order.status === 'pending' before cancelling (shop may have already accepted — do NOT cancel accepted orders) - On cancel: update status to 'auto_cancelled', trigger Cashfree refund for non-COD orders, send FCM to customer, degrade shop response_score 5. backend/src/socket/orderRoom.js - joinOrderRoom(socket, orderId): add socket to room order:{orderId} - emitStatusUpdate(orderId, status, data): broadcast to all room members - All status changes must emit to Socket.IO room in addition to DB write 6. backend/src/routes/orders.js - POST /orders: MOST IMPORTANT — server-side price calculation ONLY * Fetch product prices from Supabase by product IDs (NEVER trust client prices) * Lock stock: decrement immediately on order creation * Check idempotency key before creating * Queue notifyShop job immediately after creation * Queue autoCancel job with delay = ORDER_ACCEPT_WINDOW_SECONDS - GET /orders: list (customer sees own, shop owner sees shop's orders) - GET /orders/:id: full detail with items - PATCH /orders/:id/accept: update status, cancel autoCancel job, queue assignDelivery - PATCH /orders/:id/reject: update status, restore stock, trigger refund - PATCH /orders/:id/ready: notify delivery partner to pickup - PATCH /orders/:id/cancel: only allowed in status pending/accepted, restore stock, trigger Cashfree refund for non-COD Non-negotiable rules: - NEVER use client-sent price — ALWAYS recalculate from DB - ALWAYS check idempotency key BEFORE creating order or payment - autoCancel job: check status before cancelling — race condition exists - Order cancellable ONLY in: pending, accepted — blocked after picked_up - Stock lock happens at order creation, not at payment Write tests in backend/src/routes/orders.test.js: - POST /orders ignores client-sent price, calculates from DB - Duplicate idempotency key returns existing order, not a new one - autoCancel does NOT fire if order was accepted before 3 minutes - PATCH /cancel returns ORDER_NOT_CANCELLABLE if status is picked_up Mark tasks 3.1–3.14 as ✅ in SPRINT_PLAN.md when done.
